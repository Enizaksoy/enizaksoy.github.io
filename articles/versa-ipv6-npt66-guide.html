🔄 NPT-66 Translation Mapping

Each inside /64 maps to a unique outside /64. Host portion is preserved (stateless 1:1).

100 2009:dde8:c0fd:100::/64 → 2009:dde8:c0fd:200::/64 IPv6-NPT66-VLAN1-Pool IPv6-NPT66-VLAN1-Rule
101 2009:dde8:c0fd:101::/64 → 2009:dde8:c0fd:201::/64 IPv6-NPT66-VLAN2-Pool IPv6-NPT66-VLAN2-Rule
102 2009:dde8:c0fd:102::/64 → 2009:dde8:c0fd:202::/64 IPv6-NPT66-VLAN3-Pool IPv6-NPT66-VLAN3-Rule
💡
Example: Host 2009:dde8:c0fd:101::10 translates to 2009:dde8:c0fd:201::10 — the ::10 host portion stays the same.

🔗 Paired TVI Interfaces (Split-Tunnel between VRFs)

The FlexVNF uses paired TVI (Tunnel Virtual Interface) interfaces to route traffic between the Enterprise VRF (LAN side) and the INET-1-Transport-VR (WAN side). This is how split-tunnel / DIA (Direct Internet Access) traffic crosses VRF boundaries internally.

Architecture

LAN Side FlexVNF Internal WAN Side -------- ---------------- -------- ae0.0 / ae0.101 / ae0.102 Paired TVI Interfaces vni-0/0.0 (LAN-ZONE) (INET-1 Network) | | v v +--------------------+ +----------------------------+ +--------------------+ | Enterprise VRF | | TVI Pair (L3 link) | | INET-1-Transport-VR| | | | | | | | tvi-0/3.0 -------+-------+-- 169.254.0.3/31 -------+-------+ tvi-0/2.0 | | (LAN-VR side) | | 169.254.0.2/31 | | (WAN-VR side) | | | | | | | +--------------------+ +----------------------------+ +--------------------+ Traffic flow: Client --> ae0.x --> Enterprise VRF --> tvi-0/3.0 --> tvi-0/2.0 --> INET-1-Transport-VR --> vni-0/0.0 --> Internet 169.254.0.3 169.254.0.2

TVI Interface Configuration

! Enterprise VRF side (LAN VR)
set interfaces tvi-0/3 unit 0 family inet address 169.254.0.3/31
set routing-instances CUSTOMER-ORG-Enterprise interfaces tvi-0/3.0

! INET-1-Transport-VR side (WAN VR)
set interfaces tvi-0/2 unit 0 family inet address 169.254.0.2/31
set routing-instances INET-1-Transport-VR interfaces tvi-0/2.0

IPv6 Static Route via Split-Tunnel (Enterprise VRF)

For NPT-66 to work, the Enterprise VRF must know how to reach the translated pool prefix. IPv6 next-hop uses IPv4-mapped IPv6 address format ::ffff:x.x.x.x:

! Route translated /56 pool range toward INET-1-Transport-VR via TVI pair
set routing-instances CUSTOMER-ORG-Enterprise routing-options static route 2009:dde8:c0fd:200::/56 ::ffff:169.254.0.2

! This tells Enterprise VRF:
!   "To reach the NAT pool prefix 2009:dde8:c0fd:200::/56,
!    send traffic to 169.254.0.2 (tvi-0/2.0 in INET-1-Transport-VR)"
⚠
Critical: Without this route in Enterprise VRF, NAT never matches. Sessions show natted: No, forward-egress-interface: n/a, all packets dropped silently.

IPv6 Next-Hop Format Explained

FormatExampleMeaning
IPv4-mapped IPv6 ::ffff:169.254.0.2 IPv6 static route using IPv4 next-hop 169.254.0.2
Why needed TVI pair only has IPv4 addresses (169.254.0.x/31). IPv6 routes need this mapping to traverse IPv4-only internal links
Suffix none ... ::ffff:169.254.0.2 none The none keyword means no specific outgoing interface; resolved by routing table

Full IPv6 Routing Summary

! Enterprise VRF static routes
set routing-instances CUSTOMER-ORG-Enterprise routing-options static route 2009:dde8:c0fd:200::/56 ::ffff:169.254.0.2

! What happens at each hop:
!  1. Client sends to 2009:dde8:c0fd:200::2 (upstream router)
!  2. Enterprise VRF matches :200::/56 static route
!  3. Forwards to 169.254.0.2 via tvi-0/3.0 -> tvi-0/2.0
!  4. Arrives in INET-1-Transport-VR
!  5. CGNAT engine performs NPT-66 translation (source prefix swap)
!  6. Packet exits vni-0/0.0 toward upstream router

🛣 Upstream Router Static Routes (Recommended)

The upstream router must have /64 static routes for each translated pool, pointing to the FlexVNF WAN IP as next-hop.

! On upstream router (e.g., Cisco IOS-XE)
ipv6 route 2009:DDE8:C0FD:200::/64 2009:DDE8:C0FD:200::1
ipv6 route 2009:DDE8:C0FD:201::/64 2009:DDE8:C0FD:200::1
ipv6 route 2009:DDE8:C0FD:202::/64 2009:DDE8:C0FD:200::1
✔
Why this works: The /64 static route is more specific than the /56 connected route on the upstream router. Longest prefix match wins — the router forwards to FlexVNF instead of doing NDP on the wire.

Verification on upstream router:

Router# show ipv6 route 2009:DDE8:C0FD:201::10

Routing entry for 2009:DDE8:C0FD:201::/64        <-- /64 static wins
  Known via "static", distance 1, metric 0
  Route count is 1/1, share count 0
    via 2009:DDE8:C0FD:200::1                     <-- forwards to FlexVNF

🚀 Traffic Flow — Step by Step

Outbound: Client → Internet (through NPT-66)

1
Client sends: src=2009:dde8:c0fd:101::10 → dst=2009:dde8:c0fd:200::2
2
FlexVNF ae0.101 receives in LAN-ZONE, Enterprise VRF looks up destination
3
Static route :200::/56 via ::ffff:169.254.0.2 — forwards through paired TVI to INET-1-Transport-VR
4
NPT-66 rule matches: source :101::/64 → translates to :201::10
5
Packet exits vni-0/0.0: src=2009:dde8:c0fd:201::10 → dst=2009:dde8:c0fd:200::2 ✔

Return: Internet → Client (with static /64 routes on upstream)

1
Upstream router replies to 2009:dde8:c0fd:201::10
2
Route lookup: :201::/64 static (/64) beats :200::/56 connected (/56) — longest match wins
3
Forwards to next-hop :200::1 (FlexVNF) — NDP already resolved for this address
4
FlexVNF reverse-translates: dst=:201::10 → dst=:101::10
5
Packet delivered to Client via ae0.101 ✔

Return: WITHOUT static routes (broken — NDP failure)

1
Upstream router replies to 2009:dde8:c0fd:201::10
2
Route lookup: only :200::/56 connected on Gig3.20
3
Router thinks :201::10 is on the local segment — sends NDP Neighbor Solicitation on Gig3.20
4
Nobody answers NDP — :201::10 is a virtual translated address, no real device has it ✘

🔐 Proxy NDP — Alternative Approach

Proxy NDP allows FlexVNF to respond to Neighbor Solicitation (NS) requests on behalf of translated IPv6 addresses. When the upstream router sends an NS for a translated address (e.g., 2009:dde8:c0fd:201::10), FlexVNF replies with its own MAC address.

Configuration

! On FlexVNF WAN interface - one entry per translated /64 pool
set interfaces vni-0/0 unit 0 family inet6 proxy-ndp address [
    2009:dde8:c0fd:200::/64
    2009:dde8:c0fd:201::/64
    2009:dde8:c0fd:202::/64
]

When Proxy NDP Might Be Needed

ScenarioStatic RoutesProxy NDP
You control the upstream router Use this Not needed
ISP-managed CPE (can't add routes) Not possible Only option
Dedicated point-to-point WAN link Preferred Safe (no other devices)
Shared L2 segment with other devices Use this Risk of NDP conflict

Platform Limitations

⚠
Proxy NDP prefix restrictions:
  • Only prefix length >= /64 supported (cannot use /56 or shorter)
  • Bug ID 140313 filed for /56 support — pending engineering
  • Maximum 16 proxy-ndp entries per interface
  • If customer has >16 VLANs, proxy NDP alone cannot cover all pools

Workaround: Multiple /64 entries for /56 coverage

If proxy NDP for /56 is needed, split into individual /64 entries (max 16):

    2009:dde8:c0fd:201::/64
    2009:dde8:c0fd:202::/64
    2009:dde8:c0fd:203::/64

! Maximum 16 entries per interface - cannot cover full /56 (256 x /64)

⚠ Why Proxy NDP is NOT Recommended

✘ Proxy NDP Approach (Risky)

  • FlexVNF answers NS for every address in the /64 range
  • If real devices exist on the same L2 segment within that /64, you get duplicate NA responses
  • Traffic may black-hole or split unpredictably
  • Proxy NDP only supports >= /64 prefix (no /56 support — Bug 140313)
  • Maximum 16 proxy-ndp entries per interface
  • Difficult to troubleshoot — intermittent traffic loss

✔ Static Route Approach (Recommended)

  • /64 static route beats /56 connected via longest prefix match
  • No NDP conflict with existing devices
  • No dependency on proxy NDP feature enhancement
  • Standard routing — works on any upstream router
  • Or use a single summary: /56 route via FlexVNF if transit link uses separate prefix
  • Scales to any number of VLANs

NDP Conflict Scenario Explained

1
FlexVNF has proxy-ndp for :200::/64. A real server at :200::50 also exists on the same L2 segment
2
Upstream router sends NS: "Who has :200::50?"
3
Both respond: Real server (MAC: aa:bb:cc:...) AND FlexVNF (MAC: 52:54:00:...) send Neighbor Advertisement
4
Result: Upstream router's NDP table flaps between two MACs. Traffic to :200::50 randomly goes to either device ✘

📋 IPv4 NAPT-44 vs IPv6 NPT-66

Aspect NAPT-44 (IPv4) NPT-66 (IPv6)
Pool type egress-network (interface IP) address-list (explicit prefix)
Translation Many-to-one (PAT) One-to-one (prefix swap)
Stateful? Yes No (stateless)
Proxy ARP/NDP needed? No (translates to interface's own IP) No, if static routes used on upstream
Port translation Yes No
Return path Automatic (interface IP is known) Requires static routes on upstream
filtering-type / mapping-type Supported Not applicable (stateless)
Enterprise VRF route Usually auto-provisioned Manual static route via paired TVI needed

🕷 Common Errors & Fixes

❌
Error: "Egress network can only be configured with napt-44 or twice-napt-44"
Cause: Used egress-network in NPT-66 pool
Fix: Use address-list instead of egress-network
❌
Error: "IPv6 prefix length should be >= 64"
Cause: Tried to use /56 prefix in proxy-ndp
Fix: Use multiple /64 entries (max 16) or use static routes on upstream instead
❌
Error: "element does not exist" for filtering-type/mapping-type
Cause: These options don't exist for npt-66 (stateless)
Fix: Simply omit them — npt-66 has no port mapping options
⚠
Issue: Session shows natted: No, all packets dropped
Cause: No IPv6 route in Enterprise VRF for the destination /56 pool range
Fix: Add route 2009:dde8:c0fd:200::/56 ::ffff:169.254.0.2 in Enterprise VRF via paired TVI
⚠
Issue: Config lost between SSH sessions
Cause: Versa CLI uses private config mode per SSH session
Fix: Always commit before disconnecting
⚠
Issue: VLAN100 pool fails while VLAN101/102 work
Cause: Pool :200::/64 overlaps with the connected transit link /64 on the upstream router. Static route /64 = connected /64 → connected wins (AD 0 vs AD 1)
Fix: Use a different /64 for the transit link or for the VLAN100 pool. Ensure translated pools don't share the same /64 as the point-to-point link

💡
VLAN100 failure explained: The translated pool :200::/64 is the SAME prefix as the connected transit link. A static route for :200::/64 cannot override the connected :200::/64 — same prefix length means connected always wins (AD 0 vs AD 1). Solution: use a different /64 for the transit link or for the VLAN100 pool.

🎓 Key Lessons & Best Practices

#LessonDetail
1 Use static routes, not proxy NDP Static /64 routes on upstream router are safer and don't conflict with live devices on the L2 segment
2 Don't overlap pool prefix with transit link If pool /64 = connected /64 on upstream, static route can't override connected. Use separate transit prefix
3 Enterprise VRF needs a route via paired TVI Without route :200::/56 via ::ffff:169.254.0.2 (split-tunnel), traffic never reaches CGNAT engine
4 npt-66 = address-list only egress-network is only for napt-44. NPT-66 requires explicit pool prefix via address-list
5 Per-VLAN pools prevent collision Each VLAN gets its own /64 pool → no host-portion overlap between VLANs
6 npt-66 is stateless No filtering-type, no mapping-type, no port translation. Pure prefix-to-prefix swap (RFC 6296)
7 Commit before disconnect Versa ConfD CLI uses private config per SSH session — uncommitted changes are lost
8 Proxy NDP limits Only prefix >= /64 supported (Bug 140313 for /56). Max 16 entries per interface. Risk of NDP conflicts on shared L2 segments
9 IPv4-mapped IPv6 next-hop Use ::ffff:x.x.x.x format for IPv6 static routes via IPv4-only paired TVI interfaces
