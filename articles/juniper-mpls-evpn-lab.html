Juniper MPLS/EVPN Lab - Cisco-to-Juniper Migration

IS-IS, LDP, MPLS, iBGP VPNv4+EVPN, BFD, VRF - Juniper vMX 23.2 on CML

What This Lab Demonstrates

This lab replicates a service provider production network being migrated from Cisco IOS-XR to Juniper vMX. It consists of 8 vMX routers (4 P core + 4 PE edge) connected via full-mesh LAG bundles with LACP, running the complete MPLS/VPN protocol stack required for mobile operator services.

Key Design Points:
  • 8 routers: P1, P2, P3, P4 (core) + PE1, PE2, PE3, PE4 (edge)
  • 16 ae bundles with LACP active, 2 member GE links each
  • 40 VRFs on PE routers (20 production + 20 lab)
  • IS-IS Level 2 with wide metrics as the IGP (not OSPF)
  • BFD on both IS-IS and BGP for sub-second failure detection

IS-IS Underlay IGP

IS-IS Level 2 with wide metrics provides IP reachability between all P and PE router loopbacks. Point-to-point on all ae interfaces with BFD liveness detection (300ms x3).

LDP Label Distribution

LDP with IGP synchronization (holddown 30s) automatically assigns and distributes MPLS labels for all IS-IS-learned prefixes. Creates the Label Switched Paths (LSPs) through the core.

MPLS Label Switching

MPLS forwards packets based on labels instead of IP lookups. The transport label (from LDP) gets packets across the core; the VPN label (from BGP) identifies the VRF at the egress PE.

BGP VPNv4 + EVPN

iBGP full mesh (AS 65000) with two address families: inet-vpn unicast (VPNv4) for L3VPN services and evpn signaling for EVPN. BFD 1000ms x3 on all sessions.

VRF Service Isolation

40 VRF instances on each PE router: 20 production VRFs (CHU, HSS, NAT, INTERNET, etc.) and 20 lab VRFs (VRF10-VRF29). dot1q sub-interfaces on ge-0/0/6 and ge-0/0/7.

BFD + LACP

BFD provides sub-second failure detection on IS-IS (300ms) and BGP (1000ms). LACP active mode on all 16 ae bundles with 2 member links each for link-level redundancy.

Packet Flow: CE to CE via MPLS Core

End-to-end packet flow through the MPLS/VPN network:

  1. CE sends a packet destined for the remote site via its PE gateway
  2. Ingress PE receives it on a VRF interface, performs a VRF routing table lookup
  3. BGP VPNv4 has advertised the remote prefix with a VPN label from the egress PE
  4. Ingress PE pushes two labels: transport label (LDP, to reach egress PE loopback) + VPN label (BGP, to identify the VRF)
  5. Packet traverses the MPLS core (P routers) - each P router swaps the transport label. Penultimate hop performs PHP (pops transport label)
  6. Egress PE pops the VPN label, looks up in the correct VRF, forwards to the CE

IS-IS Underlay IGP

What is IS-IS?

Intermediate System to Intermediate System (IS-IS) is a link-state routing protocol that operates directly over Layer 2 (not over IP like OSPF). It uses ISO CLNS addressing and is the preferred IGP in large service provider networks due to its stability, scalability, and protocol-independent nature.

Why IS-IS instead of OSPF?
Service providers typically prefer IS-IS over OSPF for several reasons:
  • Stability: IS-IS runs on Layer 2, so IP routing table changes don't affect IS-IS adjacencies
  • Scalability: IS-IS handles large LSDBs better and has simpler flooding behavior
  • Extensibility: IS-IS TLV structure makes it easy to add new address families (IPv6, MPLS-TE)
  • Industry standard: Most Tier-1 carriers and mobile operators use IS-IS

IS-IS Level 2 with Wide Metrics

Level 2 Only

In this lab, all routers run IS-IS Level 2 only with wide-metrics-only. Level 2 is the "backbone" level in IS-IS, equivalent to OSPF Area 0. Since the entire network is a single flat domain, Level 1 is not needed.

Wide Metrics

Wide metrics extend the metric range from 0-63 (narrow) to 0-16777215 (wide). This is essential for modern networks with many links and for MPLS Traffic Engineering. Configured via level 2 wide-metrics-only.

Point-to-Point on AE Interfaces

All ae interfaces are configured as point-to-point. This avoids DIS (Designated Intermediate System) elections on broadcast segments, resulting in faster convergence and simpler adjacency management.

BFD on IS-IS

BFD liveness detection is enabled on every IS-IS interface with minimum-interval 300 and multiplier 3. This provides sub-second failure detection (900ms worst case) instead of waiting for IS-IS hold-time expiry.

ISO NET Addressing

ISO NET format: 49.0001.{system-id}.00
  • 49 = AFI (Authority and Format Identifier) - private addressing
  • 0001 = Area ID - all routers in the same area
  • {system-id} = 6 bytes derived from loopback IP (padded): e.g., 10.255.0.1 → 0102.5500.0001
  • 00 = SEL (NSAP Selector) - always 00 for NET

How IS-IS is Configured

Example from P1 - all ae interfaces + loopback:

IS-IS - P1 
protocols {
    isis {
        level 2 wide-metrics-only;        /* Extended metrics for modern networks */
        interface lo0.0 {
            passive;                       /* Advertise loopback, no Hellos */
        }
        interface ae0.0 {
            point-to-point;                /* No DIS election on P2P links */
            bfd-liveness-detection {
                minimum-interval 300;      /* 300ms BFD interval */
                multiplier 3;               /* Detect failure in 900ms */
            }
        }
        interface ae1.0 { point-to-point; bfd-liveness-detection { minimum-interval 300; multiplier 3; } }
        interface ae2.0 { point-to-point; bfd-liveness-detection { minimum-interval 300; multiplier 3; } }
        interface ae3.0 { point-to-point; bfd-liveness-detection { minimum-interval 300; multiplier 3; } }
        interface ae4.0 { point-to-point; bfd-liveness-detection { minimum-interval 300; multiplier 3; } }
    }
}

LDP Label Distribution Protocol

What is LDP?

Label Distribution Protocol (LDP) automatically creates MPLS label bindings for IP prefixes learned from the IGP (IS-IS in our case). It allows routers to agree on which label to use for forwarding packets toward a destination, building the Label Switched Paths (LSPs) through the core.

LDP in simple terms:
Think of labels as "express lane numbers" on a highway. Instead of reading the full destination address at every intersection (IP lookup), each router just reads the lane number (label) and forwards the packet. LDP is how routers agree on which lane number to use for each destination.

LDP with IGP Synchronization

Why IGP Sync Matters

Without IGP synchronization, there's a dangerous window after IS-IS converges but before LDP has established label bindings. During this window, traffic could be black-holed: IS-IS says "send traffic this way" but LDP hasn't yet assigned labels for that path.

IGP sync with holddown 30s tells IS-IS to advertise a maximum metric (overload) on an interface until LDP has converged on that interface (or until the holddown timer expires). This prevents traffic from being sent down a path without labels.

How LDP is Configured

LDP is enabled on every ae interface with IGP synchronization. Example from P1:

LDP - P1 
protocols {
    ldp {
        igp-synchronization {
            holddown-interval 30;       /* Wait up to 30s for LDP to converge */
        }
        interface ae0.0;               /* to P2 */
        interface ae1.0;               /* to PE1 */
        interface ae2.0;               /* to PE2 */
        interface ae3.0;               /* to P3 */
        interface ae4.0;               /* to P4 */
    }
}

Three Things Required for LDP

1. family mpls on interface

Enables MPLS encapsulation on the interface. Without this, the interface cannot send/receive labeled packets.

2. protocols mpls

Enables the MPLS data plane on the interface. Allows the router to process MPLS-labeled packets.

3. protocols ldp

Enables LDP on the interface. Starts Hello/session establishment and label exchange with neighbors.

MPLS Multiprotocol Label Switching

What is MPLS?

Multiprotocol Label Switching (MPLS) is a forwarding mechanism that uses short labels (instead of long IP addresses) to make forwarding decisions. It sits between Layer 2 and Layer 3 - sometimes called "Layer 2.5".

Why MPLS matters:
  • VPN services: MPLS labels identify which VRF a packet belongs to - foundation of L3VPN and EVPN
  • Any-to-any connectivity: Once the MPLS core is built, you can run L3VPN, L2VPN, VPLS, EVPN on top
  • Traffic Engineering: RSVP-TE can create explicit paths through the network
  • Scalability: P routers only need to know labels, not customer routes

MPLS Configuration

MPLS must be enabled on every ae interface. Example from P1:

MPLS - P1 
protocols {
    mpls {
        interface ae0.0;     /* to P2 */
        interface ae1.0;     /* to PE1 */
        interface ae2.0;     /* to PE2 */
        interface ae3.0;     /* to P3 */
        interface ae4.0;     /* to P4 */
    }
}

The MPLS Label Stack in L3VPN

A VPN packet carries two labels through the core:

  │  Ethernet Header                                      │

  │  Transport Label (LDP)  │ EXP │ S=0 │ TTL │          │
  │  "How to reach egress PE loopback"                   │
  ├──────────────────────────────────────────────────────┤
  │  VPN Label (BGP)        │ EXP │ S=1 │ TTL │          │

BGP VPNv4 + EVPN

Why BGP for VPN?

BGP (Border Gateway Protocol) is used in this lab as the VPN route distribution mechanism. We use two address families:

  • inet-vpn unicast (VPNv4) - carries L3VPN customer routes between PE routers
  • evpn signaling - carries EVPN type-2/type-5 routes for L2/L3 overlay services
BGP is the glue between VRFs:
Without BGP, each PE router's VRF would be an isolated routing table. BGP VPNv4 distributes VRF routes across the MPLS core, adding:
  • Route Distinguisher (RD): Makes customer prefixes unique in BGP
  • Route Target (RT): Controls which VRFs import/export which routes
  • VPN Label: Tells the egress PE which VRF to place the packet into
  • Next-hop: The advertising PE's loopback (resolved via IS-IS + LDP)

RD and RT Explained

Route Distinguisher (RD)

Makes prefixes unique in BGP.

If two VRFs use the same IP subnet, the RD makes them distinguishable in the BGP VPNv4 table. RD is prepended to create unique VPNv4 prefixes:

65000:5:10.10.10.0/24  /* VRF10 from PE1 */
65000:7:10.10.10.0/24  /* VRF10 from PE3 */

RD is per-PE, per-VRF - each PE uses a different RD for the same VRF. This helps with multipath and troubleshooting.

Route Target (RT)

Controls route import/export between VRFs.

When a VRF exports a route, it attaches the RT as a BGP extended community. Remote VRFs only import routes matching their RT policy:

/* PE1 exports VRF10 route with: */
vrf-target target:10:10

/* PE3 imports routes with RT 10:10 */
/* into its VRF10 instance */

Same RT = same VPN. All PEs with VRF10 and RT 10:10 exchange routes.

vrf-table-label

What is vrf-table-label?
In Junos, vrf-table-label allocates a single MPLS label per VRF (per-table label) instead of a label per prefix (per-prefix label). This is more scalable when you have many prefixes in a VRF. The label maps directly to the VRF routing table on the egress PE.

BFD Bidirectional Forwarding Detection

What is BFD?

BFD (Bidirectional Forwarding Detection) is a lightweight protocol designed for rapid failure detection. It runs independently of the routing protocols it serves (IS-IS, BGP) and provides sub-second notification when a forwarding path fails.

Why BFD?
Without BFD, failure detection relies on routing protocol timers:
  • IS-IS: Default hold-time is 30 seconds (3x 10s hello). With BFD at 300ms x3, detection drops to 900ms.
  • BGP: Default hold-time is 90 seconds (3x 30s keepalive). With BFD at 1000ms x3, detection drops to 3 seconds.
In a mobile operator network, sub-second convergence is critical for voice and real-time services.

BFD on IS-IS (300ms x3)

BFD is configured per-interface under the IS-IS protocol stanza. When the BFD session goes down, IS-IS immediately tears down the adjacency without waiting for the IS-IS hold timer.

BFD on IS-IS 
protocols {
    isis {
        interface ae0.0 {
            point-to-point;
            bfd-liveness-detection {
                minimum-interval 300;          /* Send BFD packets every 300ms */
                multiplier 3;                   /* Declare failure after 3 missed (900ms) */
            }
        }
    }
}

BFD on BGP (1000ms x3)

BFD for BGP is configured at the group level. Since BGP peers over loopback addresses (multi-hop), we use session-mode automatic which lets Junos determine single-hop vs multi-hop automatically.

BFD on BGP 
protocols {
    bgp {
        group IBGP {
            bfd-liveness-detection {
                minimum-interval 1000;          /* 1000ms interval for multi-hop BFD */
                multiplier 3;                    /* Declare failure after 3 missed (3s) */
                session-mode automatic;          /* Auto single/multi-hop detection */
            }
        }
    }
}
Why different intervals?
IS-IS BFD runs at 300ms because it monitors directly-connected links (single hop). BGP BFD runs at 1000ms because it's multi-hop (loopback-to-loopback through the core), and aggressive multi-hop BFD can cause false positives due to intermediate link congestion.

LACP Link Aggregation Control Protocol

What is LACP?

LACP (Link Aggregation Control Protocol, IEEE 802.3ad) dynamically negotiates and maintains link aggregation groups (LAGs). In Junos, LAGs are called ae (aggregated Ethernet) interfaces. LACP ensures both sides agree on which physical links belong to the bundle and monitors link health.

Design in this lab:
  • 16 ae bundles total across all 8 routers
  • 2 member GE links per bundle
  • LACP active mode on all bundles (both sides initiate negotiation)
  • P routers: 5 ae bundles (ae0-ae4) using ge-0/0/0 through ge-0/0/9
  • PE routers: 3 ae bundles (ae0-ae2) using ge-0/0/0 through ge-0/0/5
